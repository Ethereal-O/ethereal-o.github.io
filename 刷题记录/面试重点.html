1.2 手撕

1.2.1 手写线程池 (Thread Pool)

  • 必须支持 std::future 获取返回值（考察 std::package_task 和 std::bind）。

  • 必须优雅退出（Graceful Shutdown）。

  • 加分项：支持“任务优先级”，支持“任务窃取 (Work Stealing)”（类似 Go runtime 调度）。

  • 如何正确处理 condition_variable 的虚假唤醒？你的任务队列锁竞争严重吗？

1.2.2 实现一个无锁队列 (Lock-free Queue)

  • 单生产者单消费者 (SPSC) 或多生产者多消费者 (MPMC)。

  • 使用 std::atomic，理解 CAS (compare_exchange_weak)，解决 ABA 问题（虽然用指针通常不用管，但要能说出来），以及 Memory Order 的正确使用（Acquire/Release）。

1.2.3 延迟任务调度器 (Delayed Scheduler / Timer Wheel)

  • 基础版：最小堆 (Priority Queue) + 一个后台线程不断 sleep/wait。

  • 进阶版：时间轮 (Hashed and Hierarchical Timing Wheels)。这是 Linux 内核和 Netty 都在用的算法，AI 训练中的超时检测非常需要这个。

    • 分级时间轮盘

1.2.4 读写锁 (RWLock) 的写优先实现

  • 注意写出 Write-Preferred 的逻辑，防止写饥饿。

1.2.5 多维背包问题 / Bin Packing 变种

  • 给定 N 个任务，每个任务需要 [CPU, Mem, GPU] 三种资源，给定 M 个节点，每个节点资源有限。求最少需要多少节点，或者能否装下。

  • 这是一个 NP-Hard 问题，不要求最优解，要求启发式算法。

  • First Fit / Best Fit / Worst Fit 的实现。

  • 点积排序 (Dot Product)：AI 调度常用技巧，向量夹角越小，资源互补性越好（例如一个任务吃 CPU，一个吃 Mem，它俩这就很互补）。

1.2.6 带有亲和性 (Affinity) 的任务调度

  • 任务 A 和任务 B 必须调度到同一台机器（Affinity），或者不能在同一台机器（Anti-Affinity）。

  • 回溯法 (Backtracking) 或 并查集 (Union Find) 预处理分组。

1.2.7 能够满足 Gang Scheduling (全家桶调度) 的资源查找

  • 大模型训练需要 100 个 GPU 同时就绪才能跑。只要有 1 个不满足，这 100 个都不能占坑。

  • 实现 try_allocate(num_gpus)，要求原子性地成功或失败。

1.2.8 拓扑排序 (Topological Sort) + 环检测

  • 对应 LeetCode：210. 课程表 II

  • 场景：检测任务依赖中是否有死锁（环）。

  • 要求并行度分析：给定 DAG，假设有无限资源，跑完所有任务的最短时间是多少？（其实就是求关键路径 Critical Path）。

  • 如果只有 K 个 worker，跑完的时间是多少？（这是经典的 Job Shop Scheduling 问题，通常用贪心或模拟）。

1.2.9 死锁检测 (Deadlock Detection)

  • 场景：模拟操作系统资源分配图。

  • 解法：构建资源分配图（RAG），使用 DFS 寻找环。

1.2.10 LRU Cache / LFU Cache

  • 对应 LeetCode：146. LRU 缓存

  • 要求：O(1) 实现。

  • Scheduler 关联：vLLM 中的 Block Manager 或 PagedAttention 的页表置换算法核心就是 LRU 的变种。

1.2.11 接雨水 (Trapping Rain Water) / 天际线问题 (The Skyline Problem)

  • 对应 LeetCode：42. 接雨水, 218. 天际线问题

  • 关联：天际线问题非常像时间轴上的资源占用情况合并。

1.2.12 寻找两个正序数组的中位数

  • 对应 LeetCode：4. 寻找两个正序数组的中位数

  • 考点：二分查找的极致应用。

3.5.1 k8s调度

  1. karmada集群间调度器，类似k8s集群调度器

  2. k8s集群调度器

    1. Prepare【本身indexer存在一层cache】【这里会每次新建一个Snapshot cache，通过Incremental Update方式】

      1. 所有未调度的 Pod 都在这里排队。队列分为：

        • ActiveQ: 等待调度的 Pod。

        • BackoffQ: 调度失败过，等待重试的 Pod（指数退避）。

        • UnschedulableQ: 暂时无法调度的 Pod（例如资源不足），直到集群状态发生变化（如新增了 Node）才会被移回 ActiveQ。

    2. filter

      1. prefilter

        1. 快速失败，提前计算pod依赖的其他pod的信息

        2. 可以用来写gang scheduling

      2. filter

        1. 过滤可用的node

      3. postfilter

        1. 如果没有可用的，那么执行抢占，标记pod（Nominate），重新放入队列（抢占具有速率限制和容量限制，调度也有指数退避）

    3. score

      1. prescore

        1. 计算每个节点的一些预知信息，例如是否有镜像

      2. score

        1. 打分

      3. postscore

        1. 归一化

    4. Reserve【存在一层cache】

      1. 如果紧接着处理 Pod B（毫秒级差距），Pod B 去查 Scheduler Cache 时，看到的 Node 1 已经是扣除过资源的状态了。这就解决了连续调度的资源冲突问题。

      2. 如果后续阶段失败，必须触发 Unreserve 插件把资源加回去。

    5. Permit

      1. 可以实现组调度，但是不如prefilter来的快

    6. Binding

      1. PreBind

        1. 挂载网络等

      2. Bind（apiserver写入binding对象）

      3. PostBind
